## Architecture: `rdc` CLI Wrapper

The provider calls the `rdc` CLI binary. Renet is a low-level internal
component managed by rdc — the provider never calls renet directly.

```
Terraform Provider (Go)
  └── internal/client/rdc.go
        └── exec.Command("rdc", "--output", "json", "repo", "create", ...)
              └── rdc CLI (TypeScript)
                    └── SSH → renet on target machine
```

**Why this is the only option:**
- rdc manages everything: config, SSH keys, renet binary provisioning, vault
  building, network ID allocation, cross-machine credential resolution
- renet requires a QueueVaultV2 payload that only rdc knows how to build
- If rdc is missing a command or JSON output, the fix goes into rdc
- Same approach as Ansible modules — one consistent interface
- Pattern is battle-tested (`community.general.terraform` wraps `terraform` CLI)

**Requirements:**
- `rdc` binary must be installed on the machine running Terraform
- For CI/CD: `./run.sh rdc` can be used as the binary path in development

### Environment Variable Fallback

Every provider attribute has an env var fallback (AWS/Hetzner pattern):

| HCL attribute | Env var | Default |
|---------------|---------|---------|
| `config_name` | `REDIACC_CONFIG_NAME` | default config |
| `config_path` | `REDIACC_CONFIG_PATH` | auto-detect |
| `rdc_path` | `REDIACC_RDC_PATH` | `rdc` on PATH |

Resolution order: HCL → env var → default. This keeps `.tf` files
portable and credential-free. CI/CD sets `REDIACC_CONFIG_NAME=production`
without modifying Terraform configs.
```

## Provider Implementation (Go)

The Configure function implements the HCL → env var → default resolution chain:

```go
// internal/provider/provider.go — key Configure logic

func (p *rediaccProvider) Configure(ctx context.Context, req provider.ConfigureRequest, resp *provider.ConfigureResponse) {
    var config rediaccProviderModel
    diags := req.Config.Get(ctx, &config)
    resp.Diagnostics.Append(diags...)
    if resp.Diagnostics.HasError() {
        return
    }

    // Resolution chain: HCL → env var → default
    rdcPath := resolveString(config.RdcPath, "REDIACC_RDC_PATH", "rdc")
    configName := resolveString(config.ConfigName, "REDIACC_CONFIG_NAME", "")
    // configPath := resolveString(config.ConfigPath, "REDIACC_CONFIG_PATH", "")

    c := client.NewRdcClient(rdcPath, configName)
    if err := c.Verify(ctx); err != nil {
        resp.Diagnostics.AddError(
            "rdc not available",
            fmt.Sprintf("Could not execute rdc at %q: %s\n\nEnsure rdc is installed and accessible.", rdcPath, err),
        )
        return
    }
    resp.DataSourceData = c
    resp.ResourceData = c
}

func resolveString(hclValue types.String, envKey, defaultVal string) string {
    if !hclValue.IsNull() && !hclValue.IsUnknown() {
        return hclValue.ValueString()
    }
    if v := os.Getenv(envKey); v != "" {
        return v
    }
    return defaultVal
}
```

## RDC Client (Go)

The client has two execution methods:
- **`RunQuery()`** — for commands that return JSON (adds `--output json`, unwraps envelope)
- **`RunLifecycle()`** — for commands that stream output (no `--output json`, exit code only)

### JSON Envelope

Query commands return: `{success, command, data, errors, warnings, metrics}`.
`RunQuery()` automatically extracts the `data` field.

### Why Lifecycle Commands Skip `--output json`

Repo lifecycle commands (`repo create`, `repo up`, `repo down`, `repo delete`)
stream renet's raw stdout/stderr directly to the terminal. The `--output json`
envelope does NOT wrap this streamed output. These commands only communicate
success/failure via exit code.

```go
// internal/client/rdc.go

package client

import (
    "bytes"
    "context"
    "encoding/json"
    "fmt"
    "os/exec"
)

// Envelope is the JSON output format for rdc query commands.
type Envelope struct {
    Success  bool            `json:"success"`
    Command  string          `json:"command"`
    Data     json.RawMessage `json:"data"`
    Errors   json.RawMessage `json:"errors"`
    Warnings []string        `json:"warnings"`
    Metrics  json.RawMessage `json:"metrics"`
}

type RdcClient struct {
    BinaryPath string
    ConfigName string
}

func NewRdcClient(binaryPath, configName string) *RdcClient {
    return &RdcClient{
        BinaryPath: binaryPath,
        ConfigName: configName,
    }
}

func (c *RdcClient) Verify(ctx context.Context) error {
    _, err := c.RunQuery(ctx, "config", "show")
    return err
}

// RunQuery executes a query command with --output json and unwraps the envelope.
// Returns the 'data' field from the JSON envelope.
func (c *RdcClient) RunQuery(ctx context.Context, args ...string) (json.RawMessage, error) {
    cmdArgs := []string{"--output", "json"}
    if c.ConfigName != "" {
        cmdArgs = append(cmdArgs, "--config", c.ConfigName)
    }
    cmdArgs = append(cmdArgs, args...)

    cmd := exec.CommandContext(ctx, c.BinaryPath, cmdArgs...)
    var stdout, stderr bytes.Buffer
    cmd.Stdout = &stdout
    cmd.Stderr = &stderr

    if err := cmd.Run(); err != nil {
        return nil, fmt.Errorf("rdc %v failed: %s\n%s", args, err, stderr.String())
    }

    if stdout.Len() == 0 {
        return nil, nil
    }

    // Try to unwrap JSON envelope
    var env Envelope
    if err := json.Unmarshal(stdout.Bytes(), &env); err == nil && env.Data != nil {
        return env.Data, nil
    }

    // No envelope — return raw JSON
    var raw json.RawMessage
    if err := json.Unmarshal(stdout.Bytes(), &raw); err != nil {
        return nil, fmt.Errorf("rdc %v: invalid JSON output: %s", args, stdout.String())
    }
    return raw, nil
}

// RunLifecycle executes a lifecycle command (create, up, down, delete).
// Does NOT add --output json. Only checks exit code for success/failure.
func (c *RdcClient) RunLifecycle(ctx context.Context, args ...string) error {
    cmdArgs := []string{}
    if c.ConfigName != "" {
        cmdArgs = append(cmdArgs, "--config", c.ConfigName)
    }
    cmdArgs = append(cmdArgs, args...)

    cmd := exec.CommandContext(ctx, c.BinaryPath, cmdArgs...)
    var stderr bytes.Buffer
    cmd.Stderr = &stderr

    if err := cmd.Run(); err != nil {
        return fmt.Errorf("rdc %v failed: %s\n%s", args, err, stderr.String())
    }
    return nil
}

// Convenience methods for common operations

func (c *RdcClient) ConfigRepositories(ctx context.Context) (map[string]RepositoryConfig, error) {
    raw, err := c.RunQuery(ctx, "config", "repositories")
    if err != nil {
        return nil, err
    }
    if raw == nil {
        return nil, nil
    }
    var repos map[string]RepositoryConfig
    if err := json.Unmarshal(raw, &repos); err != nil {
        return nil, err
    }
    return repos, nil
}

func (c *RdcClient) MachineHealth(ctx context.Context, machine string) (*HealthResult, error) {
    raw, err := c.RunQuery(ctx, "machine", "health", machine)
    if err != nil {
        return nil, err
    }
    var health HealthResult
    if err := json.Unmarshal(raw, &health); err != nil {
        return nil, err
    }
    return &health, nil
}

func (c *RdcClient) MachineContainers(ctx context.Context, machine string) ([]ContainerInfo, error) {
    raw, err := c.RunQuery(ctx, "machine", "containers", machine)
    if err != nil {
        return nil, err
    }
    var containers []ContainerInfo
    if err := json.Unmarshal(raw, &containers); err != nil {
        return nil, err
    }
    return containers, nil
}

func (c *RdcClient) RepoCreate(ctx context.Context, name, machine, size string) error {
    return c.RunLifecycle(ctx, "repo", "create", name, "-m", machine, "--size", size)
}

func (c *RdcClient) RepoUp(ctx context.Context, name, machine string, mount bool) error {
    args := []string{"repo", "up", name, "-m", machine}
    if mount {
        args = append(args, "--mount")
    }
    return c.RunLifecycle(ctx, args...)
}

func (c *RdcClient) RepoDown(ctx context.Context, name, machine string, unmount bool) error {
    args := []string{"repo", "down", name, "-m", machine}
    if unmount {
        args = append(args, "--unmount")
    }
    return c.RunLifecycle(ctx, args...)
}

func (c *RdcClient) RepoDelete(ctx context.Context, name, machine string) error {
    return c.RunLifecycle(ctx, "repo", "delete", name, "-m", machine)
}

// Type definitions for JSON responses

type RepositoryConfig struct {
    RepositoryGuid string `json:"repositoryGuid"`
    Tag            string `json:"tag"`
    NetworkId      int    `json:"networkId"`
}

type HealthResult struct {
    Status string `json:"status"`
}

type ContainerInfo struct {
    Name       string `json:"name"`
    Status     string `json:"status"`
    State      string `json:"state"`
    Health     string `json:"health"`
    Repository string `json:"repository"`
}

## Concurrency Design

**Problem:** Terraform runs resource operations in parallel (default parallelism=10).
Two simultaneous `rdc repo up` on the same machine will conflict. Additionally,
config-modifying operations (`machine add`, `repo create`) increment the config
`version` field — concurrent writes cause version conflicts.

**Solution: mutexKV pattern** (proven by Google's Terraform provider):

| Lock key | Scope | Protects |
|----------|-------|----------|
| `rediacc/machine/<name>` | Per-machine | All operations targeting that machine |
| `rediacc/config` | Global | Config file writes (add/remove machine, create/delete repo) |

Operations on different machines can run in parallel. Operations on the same
machine are serialized. Config mutations are globally serialized.

The mutex is in-process only (not file locks) — sufficient because a single
Terraform process manages one state at a time.

## Timeout Design

Use `terraform-plugin-framework-timeouts` so users can override defaults per resource.

| Resource | Operation | Default Timeout | Why |
|----------|-----------|----------------|-----|
| `rediacc_machine` | Create | 20 min | setup-machine installs Docker, creates BTRFS |
| `rediacc_machine` | Delete | 5 min | remove-machine is fast |
| `rediacc_repository` | Create | 30 min | includes Docker image pull + compose up |
| `rediacc_repository` | Update | 30 min | expand or redeploy may pull images |
| `rediacc_repository` | Delete | 15 min | backup-before-destroy adds time |
| `rediacc_backup_schedule` | Create | 5 min | schedule set + push |
| `rediacc_infra` | Create | 10 min | push-infra deploys Traefik |

SSH operations should have their own lower-level timeout (60 seconds connection,
inherited by context from the resource-level timeout).

## Error Handling Strategy

The rdc error envelope includes `retryable` and `code` fields:

```
{success: false, errors: [{code: "SSH_CONNECTION_REFUSED", retryable: true, message: "...", guidance: "..."}]}
```

**Retryable errors** (use `tfresource.Retry()` with exponential backoff):
- SSH connection refused/timeout — host temporarily unreachable
- renet not ready — Docker daemon still starting after setup
- Config version conflict — another operation modified config concurrently

**Non-retryable errors** (fail immediately, surface to user):
- Authentication failure — wrong SSH key, host key mismatch
- Validation error — invalid repo name, missing size parameter
- Resource conflict — repo already exists with different config
- Permission denied — insufficient privileges on remote machine

**Unknown errors**: If `retryable` field is absent or error format is unexpected,
treat as non-retryable by default. Better to fail fast than retry forever.

## Data Safety Design

Repositories contain user data (encrypted LUKS volumes). Unlike typical
Terraform resources (VMs, DNS records), destroying a repo means irreversible
data loss.

**Guardrails:**
- All documentation examples include `lifecycle { prevent_destroy = true }`
- `backup_before_destroy` attribute: when true, Delete() runs
  `rdc backup push --to <storage>` before `rdc repo delete`
- Ansible modules require `force: true` for `state: absent`

**Import support (day-one requirement):** Import is the #1 adoption barrier.
Every resource supports import from v0.1.0:

```
terraform import rediacc_machine.web web-1
terraform import rediacc_repository.app "my-app:server-1"
terraform import rediacc_backup_schedule.daily "server-1"
```

Terraform 1.5+ import blocks (no CLI required):
```hcl
import {
  to = rediacc_repository.app
  id = "my-app:server-1"
}
```

Import reads `config repositories` + `machine containers` to populate state.

## Drift Detection Limitations

What the provider CAN detect:
- Repo exists or was deleted (via `config repositories`)
- Repo is running or stopped (via `machine containers`)
- Machine is registered or removed (via `config show`)

What the provider CANNOT detect:
- Repo volume size changed outside Terraform (no read-back command)
- Autostart changed outside Terraform (no JSON for `autostart list`)
- File content changed on remote (sync is one-way, no diff command)
- Backup schedule changed outside Terraform (depends on Phase 0 fix)

Accept these as v1.0 limitations. Document clearly. Address in future
rdc CLI improvements (add JSON output to `autostart list`, `repo status`).

### Local Development with `dev_overrides`

The recommended workflow uses `dev_overrides` — no `make install` or
`terraform init` needed after rebuilds. Faster iteration cycle:

```bash
# ~/.terraformrc (one-time setup)
cat > ~/.terraformrc <<'EOF'
provider_installation {
  dev_overrides {
    "rediacc/rediacc" = "/home/user/monorepo/console/packages/terraform/terraform-provider-rediacc"
  }
  direct {}
}
EOF

# Development workflow
cd packages/terraform/terraform-provider-rediacc
go build -o terraform-provider-rediacc   # Just rebuild, no install step

# Test in any directory — Terraform finds the binary via dev_overrides
cd /tmp/test-project
terraform plan   # Skips terraform init, uses local build directly
```

This pattern is used by most Terraform provider developers. The
`dev_overrides` section tells Terraform to use the local binary
instead of downloading from the registry. Remove `~/.terraformrc`
when switching back to released versions.

## Versioning Strategy

Start at **0.1.0**, not 1.0.0. Semantic versioning below 1.0 allows
breaking changes as the API stabilizes. Release milestones:

| Version | Scope |
|---------|-------|
| 0.1.0 | Machine + repository resources with import. Minimum viable provider. |
| 0.2.0 | Data sources (machines, containers, health, repositories) |
| 0.3.0 | Backup schedule + infra resources |
| 0.x.y | Bug fixes and UX improvements from user feedback |
| 1.0.0 | Stable API. Only after real-world usage confirms the schema is right. |

Don't wait for perfection — ship 0.1.0 with import support and iterate.
