# Подписка и лицензирование

Лицензирование в Rediacc состоит из трёх взаимосвязанных компонентов:

- `account` подписывает права и отслеживает использование
- `rdc` выполняет аутентификацию, запрашивает лицензии, доставляет их на машины и применяет их во время выполнения
- `renet` (среда выполнения на машине) проверяет установленные лицензии локально без обращения к серверу учётных записей

На этой странице объясняется, как эти компоненты работают вместе при локальных развёртываниях.

## Что делает лицензирование

Лицензирование контролирует две различные вещи:

- **Учёт доступа к машинам** через **плавающие лицензии**
- **Авторизацию среды выполнения репозитория** через **лицензии репозитория**

Они связаны между собой, но это разные артефакты.

## Как работает лицензирование

`account` является источником истины для планов, переопределений контрактов, состояния активации машин и ежемесячной выдачи лицензий репозиториев.

`rdc` работает на вашей рабочей станции. Он авторизует вас на сервере учётных записей, запрашивает необходимые лицензии и устанавливает их на удалённые машины через SSH. При выполнении команды репозитория `rdc` проверяет наличие необходимых лицензий и валидирует их на машине во время выполнения.

Типичный рабочий процесс выглядит так:

1. Вы проходите аутентификацию с помощью `rdc subscription login`
2. Вы выполняете команду репозитория, например `rdc repo create`, `rdc repo up` или `rdc repo down`
3. Если требуемая лицензия отсутствует или истекла, `rdc` запрашивает её у `account`
4. `rdc` записывает подписанную лицензию на машину
5. Лицензия проверяется локально на машине, и операция продолжается

Дополнительную информацию см. в разделе [rdc vs renet](/ru/docs/rdc-vs-renet) для понимания разделения между рабочей станцией и сервером, а также [Репозитории](/ru/docs/repositories) для описания жизненного цикла репозитория.

Для автоматизации и ИИ-агентов используйте токен подписки с ограниченной областью вместо входа через браузер:

```bash
rdc subscription login --token "$REDIACC_SUBSCRIPTION_TOKEN"
```

Вы также можете передать токен непосредственно через переменные окружения, чтобы CLI мог выдавать и обновлять лицензии репозитория без какого-либо интерактивного входа:

```bash
export REDIACC_SUBSCRIPTION_TOKEN="rdt_..."
export REDIACC_ACCOUNT_SERVER="https://www.rediacc.com/account"
```

## Слоты машин и лицензии репозиториев

### Слоты машин (на стороне сервера)

Отслеживание слотов машин применяется на стороне сервера. Когда CLI выдаёт лицензию репозитория, сервер учётных записей проверяет квоту слотов машин подписки (например, 2 машины для Community, 5 для Professional). Слот удерживается в течение 1 часа с момента последней выдачи лицензии репозитория на этой машине и автоматически освобождается после неактивности. Таким образом, план на 5 слотов может охватывать множество машин со временем, так как слоты удерживаются только во время активной подготовки.

На машине не хранится файл лицензии машины. Проверка слотов происходит на сервере во время выдачи.

### Лицензия репозитория

Лицензия репозитория — это подписанная лицензия для одного репозитория на одной машине. Это единственный файл лицензии, хранящийся на машине (`/var/lib/rediacc/license/repos/{guid}.json`).

Она используется для:

- `rdc repo create` и `rdc repo fork`, валидация перед подготовкой (предварительно выданная без доказательств идентичности, затем повторно выданная с доказательствами идентичности после создания)
- `rdc repo resize` и `rdc repo expand`, полная валидация, включая проверку срока действия
- `rdc repo up`, `rdc repo down`, `rdc repo delete`, валидация с **пропуском срока действия**
- `rdc repo push`, `rdc repo pull`, `rdc repo sync`, валидация с **пропуском срока действия**
- автозапуск репозитория при перезапуске машины, валидация с **пропуском срока действия**

Лицензии репозитория привязаны к машине и целевому репозиторию. Каждая лицензия содержит ID машины, GUID репозитория, ID подписки, ограничения плана и срок действия. Для зашифрованных репозиториев Rediacc также проверяет идентичность LUKS основного тома.

На одной машине могут сосуществовать несколько подписок. Каждый репозиторий имеет собственную лицензию со своим контекстом подписки.

## Ограничения по умолчанию

Размер репозитория зависит от уровня прав:

- Community: до `10 GB`
- платные планы: ограничение плана или контракта

Ограничения по умолчанию для платных планов:

| План | Плавающие лицензии | Размер репозитория | Ежемесячная выдача лицензий репозитория | Срок действия сертификата делегирования (по умолч. / макс.) |
|------|------|------|------|------|
| Community | 2 | 10 GB | 500 | 15d / 30d |
| Professional | 5 | 100 GB | 5,000 | 60d / 120d |
| Business | 20 | 500 GB | 20,000 | 90d / 180d |
| Enterprise | 50 | 2048 GB | 100,000 | 120d / 365d |

Ограничения, специфичные для контракта, могут повышать или понижать эти значения для конкретного клиента. Срок действия сертификата делегирования также жёстко ограничен значением `subscription.expiresAt + 3 day grace`, поэтому для ежемесячных подписок сертификаты естественным образом выравниваются с циклом выставления счётов. Полные правила см. в разделе [Цепочка лицензий и делегирование - Политика сроков действия](/ru/docs/license-chain).

## Период благодати при миграции ВМ

Когда хостинг-провайдер переносит виртуальную машину на другое физическое оборудование, ID машины меняется (он получается из идентификаторов оборудования, таких как DMI UUID, `/etc/machine-id` и MAC-адреса сетевого интерфейса). Лицензии репозитория привязаны к ID машины, поэтому при миграции все лицензии обычно становились бы недействительными.

Для прозрачной обработки этой ситуации лицензии репозитория включают **40-дневный период благодати для ID машины**. Если ID машины не совпадает, но лицензия была выдана менее 40 дней назад, лицензия всё ещё принимается. Так как лицензии обновляются каждые 30 дней, следующее обновление автоматически привяжет её к новому ID машины.

На практике:
- ВМ перенесена, ID машины изменился: репозитории продолжают работать (в пределах 40-дневного окна)
- Следующая операция `rdc` обновляет лицензию с новым ID машины
- Ручное вмешательство не требуется
- Проверьте ID машины и статус лицензии с помощью `rdc machine query --system --licenses --name <machine>`

**Пользователи канала Edge** получают удвоенные ограничения Community без дополнительной платы (репозитории размером 20 GB, 1,000 выдач в месяц, 4 машины). Платные планы доступны только в канале Stable. Дополнительную информацию см. в разделе [Каналы выпуска](/ru/docs/release-channels).

## Что происходит при создании, запуске, остановке и перезапуске репозитория

### Создание и клонирование репозитория

При создании или клонировании репозитория:

1. `rdc` убеждается, что ваш токен подписки доступен (при необходимости запускает аутентификацию по коду устройства)
2. `rdc` предварительно выдаёт лицензию репозитория с сервера учётных записей (сервер проверяет квоту слотов машин и ежемесячные ограничения выдачи на этом этапе)
3. Предварительно выданная лицензия репозитория записывается на машину и валидируется локально (подпись, ID машины, GUID репозитория, срок действия и ограничение размера)
4. После успешного создания `rdc` повторно выдаёт лицензию репозитория с доказательствами идентичности репозитория (UUID LUKS или отпечаток хранилища)

Эта выдача с поддержкой учётной записи учитывается в вашем ежемесячном использовании **выдачи лицензий репозитория**. Каждая лицензия содержит адрес электронной почты владельца учётной записи и название компании, которые логируются при валидации лицензии renet.

### Запуск, остановка и удаление репозитория

`rdc` валидирует установленную лицензию репозитория на машине, но **пропускает проверку срока действия**. Подпись, ID машины, GUID репозитория и идентичность всё равно проверяются. Пользователи никогда не теряют доступ к работе со своими репозиториями, даже с истёкшей подпиской.

### Изменение размера и расширение репозитория

`rdc` выполняет полную валидацию лицензии репозитория, включая проверку срока действия и ограничений размера.

### Перезапуск машины и автозапуск

Автозапуск использует те же правила, что и `rdc repo up`: срок действия пропускается, поэтому репозитории всегда перезапускаются свободно.

Лицензии репозитория используют долгосрочную модель валидности:

- `refreshRecommendedAt` — мягкая точка обновления
- `hardExpiresAt` — жёсткая точка истечения

Если лицензия репозитория устарела, но ещё не достигла жёсткого истечения срока, среда выполнения может продолжать работу. Как только достигается жёсткое истечение срока, `rdc` должен обновить её для операций resize и expand.

### Другие операции репозитория

Операции перечисления репозиториев, получения информации репозитория и монтирования не требуют валидации лицензии.

## Проверка статуса и обновление лицензий

Вход пользователя:

```bash
rdc subscription login
```

Вход для автоматизации или ИИ-агента:

```bash
rdc subscription login --token "$REDIACC_SUBSCRIPTION_TOKEN"
```

Для неинтерактивных окружений установка `REDIACC_SUBSCRIPTION_TOKEN` является наиболее простым вариантом. Область действия токена должна быть ограничена только операциями подписки и лицензии репозитория, необходимыми агенту.

Показать статус подписки, поддерживаемый учётной записью:

```bash
rdc subscription status
```

Показать детали активации машины для одной машины:

```bash
rdc subscription activation status -m hostinger
```

Показать детали установленной лицензии репозитория на одной машине:

```bash
rdc subscription repo status -m hostinger
```

Обновить лицензии репозитория в пакетном режиме на машине:

```bash
rdc subscription refresh repos -m hostinger
```

Репозитории, обнаруженные на машине, но отсутствующие в локальной конфигурации `rdc`, отклоняются при пакетном обновлении. Они сообщаются как ошибки и не классифицируются автоматически.

Принудительное обновление лицензии репозитория для существующего репозитория:

```bash
rdc subscription refresh repo --name my-app -m hostinger
```

При первом использовании лицензированная операция репозитория или резервного копирования, которая не находит пригодной лицензии репозитория, может автоматически инициировать передачу авторизации учётной записи. CLI печатает URL авторизации, пытается открыть браузер в интерактивных терминалах и повторяет операцию один раз после успешной авторизации и выдачи.

В неинтерактивных окружениях CLI не ожидает подтверждения браузера. Вместо этого он предлагает передать токен с ограниченной областью через `rdc subscription login --token ...` или `REDIACC_SUBSCRIPTION_TOKEN`.

Для первоначальной настройки машины см. раздел [Настройка машины](/ru/docs/setup).

## Поведение в автономном режиме и истечение срока действия

Валидация лицензии происходит локально на машине. Она не требует активного подключения к серверу учётных записей.

Это означает:

- работающему окружению не нужна активная связь с учётной записью при каждой команде
- все репозитории всегда могут запускаться, останавливаться и удаляться даже с истёкшими лицензиями — пользователи никогда не теряют доступ к работе со своими репозиториями
- операции подготовки (`create`, `fork`) требуют предварительно выданную лицензию репозитория, а операции расширения (`resize`, `expand`) требуют действующую лицензию репозитория
- действительно истёкшие лицензии репозитория должны быть обновлены через `rdc` перед операциями resize и expand
- подписи лицензии проверяются по встроенному открытому ключу; отключение проверки подписи невозможно

## Поведение при восстановлении

Автоматическое восстановление намеренно ограничено:

- `missing`: `rdc` может авторизовать доступ к учётной записи при необходимости, обновить лицензии репозитория в пакетном режиме и повторить один раз
- `expired`: `rdc` может обновить лицензии репозитория в пакетном режиме и повторить один раз
- `machine_mismatch`: немедленный отказ с предложением переиздать лицензию из текущего контекста машины
- `repository_mismatch`: немедленный отказ с предложением явно обновить лицензии репозитория
- `sequence_regression`: немедленный отказ как проблема целостности или состояния лицензии репозитория
- `invalid_signature`: немедленный отказ как проблема целостности или состояния лицензии репозитория
- `identity_mismatch`: немедленный отказ — идентичность репозитория не совпадает с установленной лицензией

Эти случаи немедленного отказа не расходуют автоматически подкрепленные учётной записью вызовы обновления или выдачи.

## Сертификаты делегирования для локального развёртывания

Для локального развёртывания и развёртываний в изолированных сетях вышестоящий сервер учётной записи выдаёт **сертификат делегирования**, который авторизует вашу локальную установку подписывать лицензии с помощью собственного ключа Ed25519. Это ограничивает вас ограничениями вашего плана и создаёт цепочку, защищённую от подделки.

Ключевые моменты для владельцев подписки:

- **Один активный сертификат на подписку.** Каждая локальная установка применяет ежемесячные и ежемашинные квоты по собственному локальному реестру, поэтому наличие нескольких установок умножало бы эффективную квоту без возможности примирения. Клиентам, которым нужны production, staging и DR, необходимо приобрести отдельную подписку для каждой установки.
- **Срок действия по умолчанию на основе уровня** (15d / 60d / 90d / 120d) и максимальные пределы (30d / 120d / 180d / 365d) — см. таблицу ограничений выше.
- **Самообслуживание через портал клиента.** Владельцы и администраторы организации могут создавать, обновлять и отзывать сертификаты делегирования по адресу `/account/delegation-certs`. Страница видна всем клиентам независимо от уровня плана — различаются только ограничения.
- **Автоматическое обновление** поддерживается через однократную начальную установку, которая выдаёт API-токен с областью `delegation:renew` для использования локальной установкой при вызовах обновления к вышестоящей системе.
- **Обновление в изолированной сети** поддерживается через подписанный манифест запроса обновления, который администратор локальной установки скачивает, передаёт офлайн на вышестоящую систему, и вышестоящая система обрабатывает для выдачи нового сертификата.

Для операционной установки см. раздел [Локальное развёртывание - Лицензирование для развёртываний в изолированных сетях](/ru/docs/on-premise), а для криптографического дизайна см. раздел [Цепочка лицензий и делегирование](/ru/docs/license-chain).

## Ежемесячная выдача лицензий репозитория

Эта метрика подсчитывает успешную деятельность по выдаче лицензий репозитория с поддержкой учётной записи в текущем UTC календарном месяце.

Включает:

- первоначальную выдачу лицензии репозитория
- успешное обновление лицензии репозитория, возвращающее вновь подписанную лицензию

Не включает:

- неизменённые пакетные записи
- неудачные попытки выдачи
- неотслеживаемые репозитории, отклонённые до выдачи

Если вам нужен обращённый к клиенту просмотр использования и истории выдачи лицензий репозитория, используйте портал учётной записи. Если вам нужна проверка на стороне машины, используйте `rdc subscription activation status -m` и `rdc subscription repo status -m`.
