# المستودعات

**المستودع** هو صورة قرص مشفرة بـ LUKS على الخادم البعيد. عند تحميله، يوفر:
- نظام ملفات معزول لبيانات تطبيقك
- خادم Docker مخصص (منفصل عن Docker المضيف)
- عناوين IP loopback فريدة لكل خدمة ضمن شبكة فرعية /26

## إنشاء مستودع

```bash
rdc repo create --name my-app -m server-1 --size 10G
```

| الخيار | مطلوب | الوصف |
|--------|-------|-------|
| `-m, --machine <name>` | نعم | الجهاز الهدف الذي سيتم إنشاء المستودع عليه |
| `--size <size>` | نعم | حجم صورة القرص المشفرة (مثل `5G`, `10G`, `50G`) |
| `--skip-router-restart` | لا | تخطي إعادة تشغيل خادم المسار بعد العملية |

ستظهر في المخرجات ثلاث قيم مُولَّدة تلقائيًا:

- **GUID المستودع** -- معرف UUID يُحدد صورة القرص المشفرة على الخادم.
- **بيانات الاعتماد** -- عبارة مرور عشوائية تُستخدم لتشفير/فك تشفير وحدة LUKS.
- **معرف الشبكة** -- عدد صحيح (يبدأ من 2816، يتزايد بمقدار 64) يحدد شبكة IP الفرعية لخدمات هذا المستودع.

> **احفظ بيانات الاعتماد بأمان.** إنها مفتاح التشفير للمستودع. إذا فُقدت، لا يمكن استرداد البيانات. تُخزَّن بيانات الاعتماد في ملف `config.json` المحلي ولا تُخزَّن على الخادم.

## التحميل وإلغاء التحميل

يقوم التحميل بفك تشفير المستودع وجعل نظام الملفات الخاص به قابلًا للوصول. يغلق إلغاء التحميل الوحدة المشفرة.

```bash
rdc repo mount --name my-app -m server-1  # فك التشفير والتحميل
rdc repo unmount --name my-app -m server-1  # إلغاء التحميل وإعادة التشفير
```

| الخيار | الوصف |
|--------|-------|
| `--checkpoint` | إنشاء نقطة تفتيش CRIU قبل التحميل/إلغاء التحميل (للحاويات ذات تسمية `rediacc.checkpoint=true`) |
| `--skip-router-restart` | تخطي إعادة تشغيل خادم المسار بعد العملية |

## التحقق من الحالة

```bash
rdc repo status --name my-app -m server-1
```

## قائمة المستودعات

```bash
rdc repo list -m server-1
```

## تغيير الحجم

تعيين المستودع إلى حجم دقيق أو التوسيع بمقدار معين:

```bash
rdc repo resize --name my-app -m server-1 --size 20G  # التعيين إلى حجم دقيق
rdc repo expand --name my-app -m server-1 --size 5G  # إضافة 5G إلى الحجم الحالي
```

> يجب أن يكون المستودع غير محمَّل قبل تغيير الحجم.

## النسخ المتفرع (Fork)

إنشاء نسخة من مستودع موجود في حالته الحالية:

```bash
rdc repo fork --parent my-app --tag staging -m server-1
```

تستخدم النسخ المتفرعة نموذج name:tag: تسمى النسخة الناتجة `my-app:staging`. يُنشئ هذا نسخة مشفرة جديدة بـ GUID ومعرف شبكة خاصين بها، مع مشاركة اسم المستودع الأصلي. تشترك النسخة المتفرعة في نفس بيانات اعتماد LUKS الخاصة بالمستودع الأصلي.

## التحقق من الصحة

التحقق من سلامة نظام الملفات للمستودع:

```bash
rdc repo validate --name my-app -m server-1
```

## الملكية

تعيين ملكية الملفات داخل المستودع للمستخدم العام (UID 7111). يكون هذا ضروريًا عادةً بعد رفع الملفات من محطة العمل، حيث تصل بـ UID المحلي الخاص بك.

```bash
rdc repo ownership --name my-app -m server-1
```

يكتشف الأمر تلقائيًا أدلة بيانات حاويات Docker (نقاط ربط قابلة للكتابة) ويستثنيها. يمنع هذا كسر الحاويات التي تدير الملفات بـ UIDs الخاصة بها (مثل: MariaDB=999, www-data=33).

| الخيار | الوصف |
|--------|-------|
| `--uid <uid>` | تعيين UID مخصص بدلًا من 7111 |
| `--skip-router-restart` | تخطي إعادة تشغيل خادم المسار بعد العملية |

لفرض الملكية على جميع الملفات بما فيها بيانات الحاويات:

```bash
rdc repo ownership --name my-app -m server-1
```


راجع [دليل الترحيل](/en/docs/migration) للاطلاع على شرح كامل لمتى وكيف تستخدم الملكية أثناء ترحيل المشروع.

## القالب

تطبيق قالب لتهيئة مستودع بملفات:

```bash
rdc repo template apply --name my-template -m server-1 -r my-app --file ./my-template.tar.gz
```

## الحذف

حذف مستودع وجميع البيانات الموجودة فيه بشكل دائم:

```bash
rdc repo delete --name my-app -m server-1
```

> يؤدي هذا إلى حذف صورة القرص المشفرة بشكل دائم. لا يمكن التراجع عن هذا الإجراء.

## ترحيل المستودع

الترحيل الحي لمستودع من جهاز إلى آخر مع أدنى توقف.

```bash
rdc repo migrate --name my-app --from server-1 --to server-2
```

| الخيار | الوصف |
|--------|-------|
| `--provision` | تجهيز المستودع على الجهاز الهدف قبل الترحيل (إنشاء صورة LUKS وتسجيل الإعدادات) |
| `--checkpoint` | إنشاء نقطة تفتيش CRIU للحاويات الجارية قبل التحويل |
| `--bwlimit <kbps>` | تحديد عرض نطاق rsync بالكيلوبايت في الثانية |
| `--skip-dns` | تخطي تحديث سجلات DNS بعد التحويل |

**تدفق ثلاثي المراحل:**

1. **النسخ المسبق الحي** - ينقل rsync البيانات بينما يظل المستودع يعمل على المصدر. تُنقل الملفات الكبيرة قبل أي توقف.
2. **التحويل** - يتوقف المستودع على المصدر، وتُزامن تمريرة rsync النهائية التغييرات المتبقية، ثم يبدأ المستودع على الهدف.
3. **البدء على الهدف** - يقوم renet بتحميل المستودع وتشغيله على الجهاز الهدف. يُحدَّث DNS إلا إذا تم تمرير `--skip-dns`.

![ترحيل مستودع حي](/img/repo-migrate-flow.svg)

**المقارنة بين push والترحيل:**

| | `repo push` | `repo migrate` |
|--|-------------|----------------|
| العملية | نسخ | نقل |
| المصدر بعد العملية | دون تغيير | متوقف |
| التوقف | لا يوجد (نسخ فقط) | نافذة تحويل قصيرة |
| تحديث DNS | لا | نعم (إلا مع `--skip-dns`) |
| حالة الاستخدام | النسخ الاحتياطي، استنساخ بيئة التطوير | استبدال الجهاز، نقل الخادم |

## التنظيف

بعد حذف المستودعات أو الاسترداد من العمليات الفاشلة، قد تبقى أدلة تحميل يتيمة وملفات قفل وعلامات غير قابلة للنقل. يزيل التنظيف هذه بأمان:

```bash
# معاينة ما سيتم حذفه
rdc machine prune --name server-1 --dry-run

# إزالة الموارد اليتيمة
rdc machine prune --name server-1
```

تتأثر فقط الموارد التي لا تمتلك صورة مستودع مطابقة. لا تُزال أدلة التحميل غير الفارغة أبدًا.
